Limited Reasoner Demo

This is demo of a limited reasoner for decidable reasoning in first-order knowledge bases.

Quick start: have a look at the "Problem Description", hit the "Click Me!" button, and check the output under "Reasoning Results".

About the Reasoner

The logical language features functions and equality, first-order quantification, standard names, and sorts. (Predicates are not built-in but can be simulated with no overhead using boolean functions.) Knowledge bases are restricted to be in clausal form with only universally quantified variables – these KBs are called proper+. (Existentially quantified variables in the knowledge base can be simulated with functions.) Queries are not subject to any syntactic restriction.

Decision procedures for sound entailment [KR-2016] and sound consistency [ECAI-2016] are implemented. Future work are the conditional beliefs [ECAI-2016] (should be easy), actions [KR-2014] [ (as a preprocessing step), and introspection [IJCAI-2013].

The reasoner is written in C++. The code is available at github.com/schwering/lela. This demo is compiled by emscripten, which compiles C++ to Javascript. The performance of the Javascript is about 10× slower than the native performance (on my computer, anyway). (Which is funny, it should be only about 2× slower according to the times reported at the emscripten website.)

Please report bugs to c.schwering@unsw.edu.au.

Problem Specification Language

A problem description consists of four segments:

  1. declarations of sorts, variables, (standard) names, and functions;
  2. the knowledge definition;
  3. optional abbreviations for formulas;
  4. entailment and/or consistency queries.

The syntax of these segments is as follows, where this font is a word the problem description language, and this font represents a meta-variable in such a word.

All sorts, variables, names, and function symbols must be declared before they are used. A declaration of a sort is of the form Sort id;, where id is the identifier of the sort. A declaration of a variable is of the form Var id -> sort-id;, where id is the identifier of the variable and sort-id is the identifier of a previously declared sort. A declaration of a (standard) name is of the form Name id -> sort-id;, where id is the identifier of the name and sort-id is the identifier of a previously declared sort. A declaration of a function symbol is of the form Fun id/arity -> sort-id;, where id is the identifier of the function symbol, arity ≥ 0 is its arity, and sort-id is the identifier of a previously declared sort. An identifier must start with a lower- or upper-case letter and may be followed by further letters, digits, or underscore symbols. Identifiers must all be distinct; for example, an identifier used for a sort cannot be used for a function symbol.

The knowledge base consists of a sequence of clauses, each of which is written as KB (literal1 || ... || literalk);, where literali denotes a literal. The sequence of literals may also be empty, that is, KB (); or there may be just one literal, that is, KB (literal1);. Every literali is of the form t1 == t2 or t1 != t2, where ti is a term. A term is either the identifier of a previously declared variable, a (standard) name, or a function symbol f applied to other terms t'i, written f(t'k, ..., t'k). When f is a constant symbol (that is, has arity 0), the brackets may be omitted. Variables in the KB are implicitly universally quantified.

For convenience, abbreviations for formulas can be defined. The syntax for that is Let id := formula;, where id is the identifier by which the formula can later be referred. The set of formulas includes all literals as defined above, negation ! formula, conjunction formula1 && formula2, disjunction formula1 || formula2, universal Fa x formula, existential Ex x formula, where x is the identifier of some previously declared variable. Expressions can be put in brackets, (formula). Conjunction binds stronger than disjunctions. The scope of unary operators, that is, negation and quantifiers, is minimal; use brackets to increase the scope. Finally, formula may also be a previously defined identifier of a formula abbreviation.

Finally, queries are of the form Entails (k, formula); or Consistent (k, formula);, where k ≥ 0 is a natural number that denotes the reasoning effort and formula is a formula as defined above. Both entailment and consistency queries are sound but incomplete with respect to classical logic. The reasoning effort is measured in case splits (a split means to select a term f(n1,...,nl) and consider all possible denotations of that term) in entailment queries, and as term assignments (an assignment means to select a term f(t1,...,tl) and fix their denotation) in consistency queries.

The lexer and parser are hand-written (recursive descent) just for this demo and hardly tested. So they're probably not very fast, and, more importantly, very buggy. So let's just hope it works ☺.
